# Design Document: Multi-File Review Output

## Overview

The multi-file review output feature transforms the zaob-review-kit (zrk) code review workflow from producing a single monolithic document to generating a structured set of Markdown files. Each file serves a specific purpose: a summary file provides overview and navigation, individual role files contain perspective-specific findings, and a verdict file consolidates recommendations and final decisions.

This design addresses the current limitation where Claude.ai web defaults to creating `.docx` files with mixed role perspectives, making navigation and focused review difficult. By splitting output into separate Markdown files with clear naming conventions and cross-linking, reviewers can efficiently focus on specific concerns (security, performance, architecture) without distraction.

The solution integrates seamlessly with the existing prep-review workflow, writing all output files to the same timestamped directory structure under `.materials/` that already contains `review_context.xml`, `review_prompt.md`, and `review.patch`.

## Architecture

### File Organization Strategy

The multi-file output uses a numbered prefix system (00-99) to ensure correct reading order when files are sorted alphabetically. This approach works across all file systems and tools without requiring special sorting logic.

```
.materials/YYYYMMDD-HHMMSS/
├── review_context.xml          # existing: generated by prep-review
├── review_prompt.md            # existing: generated by prep-review
├── review.patch                # existing: generated by prep-review (commit mode)
├── temp/                       # new: agent working directory
│   ├── file_list.txt           # cached list of files to review
│   ├── draft_findings.md       # intermediate analysis results
│   └── context_notes.md        # agent's working notes
├── 00-summary.md               # new: overview and navigation
├── 01-swe-review.md            # new: Senior Software Engineer findings
├── 02-sa-review.md            # new: Software Architect findings
├── 03-qa-review.md             # new: Quality Assurance findings
├── 04-pe-review.md             # new: Performance Engineer (conditional)
├── 05-se-review.md             # new: Security Engineer (conditional)
├── 06-oe-review.md             # new: Operations Engineer (conditional)
├── 07-ceo-review.md            # new: CEO perspective (conditional)
├── 08-devil-advocate.md        # new: Devil's Advocate (conditional)
└── 99-verdict.md               # new: recommendations and final decision
```

### Role Activation Logic

The design distinguishes between core roles (always active) and triggered roles (conditionally active based on calibration rules):

**Core Roles** (always create files):
- Senior Software Engineer (SWE)
- Software Architect (SA)
- Quality Assurance (QA)

**Triggered Roles** (create files only when calibration rules activate them):
- Performance Engineer (PE) - triggered by DB queries, async code, nested loops
- Security Engineer (SE) - triggered by auth code, new dependencies
- Operations Engineer (OE) - triggered by new endpoints, config changes, retry logic
- CEO - triggered by breaking changes, public API changes
- Devil's Advocate - triggered by breaking changes, public API changes

This activation logic is defined in the existing `review-prompting.md` calibration rules and remains unchanged.

### Navigation Model

The summary file serves as the entry point and navigation hub. It contains a table of contents with links to all generated role files and the verdict file. The verdict file links back to the summary, creating a bidirectional navigation structure. Role files may cross-reference each other when findings are related.

```
00-summary.md
    ├──> 01-swe-review.md
    ├──> 02-sa-review.md
    ├──> 03-qa-review.md
    ├──> 04-pe-review.md (if triggered)
    ├──> 05-se-review.md (if triggered)
    └──> 99-verdict.md ──> back to 00-summary.md
```

## Components and Interfaces

### Temp Directory Manager Component

Responsible for managing the agent's working directory within the materials folder.

**Interface:**
```
TempDirectoryManager {
  create_temp_directory(materials_dir: Path) -> Result<Path>
  get_temp_path(materials_dir: Path) -> Path
  write_working_file(temp_dir: Path, filename: String, content: String) -> Result<()>
  read_working_file(temp_dir: Path, filename: String) -> Result<String>
  cleanup_temp_directory(temp_dir: Path) -> Result<()>
}
```

**Responsibilities:**
- Create `.materials/<timestamp>/temp/` directory
- Provide consistent temp path for agent working files
- Cache file lists, intermediate analysis, and draft content
- Optional cleanup after review completion
- Never delete temp directory automatically (preserve for debugging)

### Documentation Collector Component

Responsible for identifying and including project documentation in review context.

**Interface:**
```
DocumentationCollector {
  find_documentation_files(project_root: Path) -> Vec<Path>
  is_documentation_file(path: Path) -> bool
  get_documentation_patterns() -> Vec<String>
  include_in_context(doc_files: Vec<Path>, context_builder: &mut ContextBuilder) -> Result<()>
}
```

**Responsibilities:**
- Search for README files in all directories
- Identify docs/, documentation/, .github/ directories
- Detect architecture diagrams (*.drawio, *.mmd, *.puml, *.svg, *.png in docs/)
- Find API specs (openapi.yaml, swagger.yaml, api-spec.md)
- Locate design documents (DESIGN.md, ARCHITECTURE.md, ADR-*.md)
- Include documentation files in review_context.xml
- Respect .archignore and .repomixignore patterns

### File Generator Component

Responsible for creating and writing review output files.

**Interface:**
```
FileGenerator {
  create_summary_file(directory: Path, content: SummaryContent) -> Result<()>
  create_role_file(directory: Path, role: Role, content: RoleContent) -> Result<()>
  create_verdict_file(directory: Path, content: VerdictContent) -> Result<()>
  ensure_directory_exists(directory: Path) -> Result<()>
}
```

**Responsibilities:**
- Create output directory if it doesn't exist
- Generate files with correct naming convention
- Handle file creation errors gracefully (log and continue)
- Preserve existing files (no modification or deletion)

### Content Formatter Component

Responsible for formatting review content as Markdown.

**Interface:**
```
ContentFormatter {
  format_summary(summary: Summary, file_walkthrough: Table, risk: RiskAssessment, toc: TableOfContents) -> String
  format_role_findings(role: Role, findings: Vec<Finding>) -> String
  format_verdict(tests: Vec<SuggestedTest>, recommendations: Vec<Recommendation>, verdict: Verdict) -> String
  format_table_of_contents(role_files: Vec<String>) -> String
  format_markdown_link(text: String, target: String) -> String
}
```

**Responsibilities:**
- Generate valid Markdown syntax
- Format tables (file walkthrough, TOC)
- Format code blocks and lists
- Generate relative links between files

### Role Activator Component

Responsible for determining which roles should be active based on calibration rules.

**Interface:**
```
RoleActivator {
  determine_active_roles(change_size: usize, change_types: Vec<ChangeType>) -> Vec<Role>
  is_core_role(role: Role) -> bool
  is_triggered_role(role: Role) -> bool
}
```

**Responsibilities:**
- Apply calibration rules from review-prompting.md
- Distinguish between core and triggered roles
- Return ordered list of active roles

### Review Memory Integrator Component

Responsible for updating review_memory.md with multi-file references.

**Interface:**
```
ReviewMemoryIntegrator {
  append_review_entry(memory_file: Path, entry: ReviewMemoryEntry) -> Result<()>
  format_files_reviewed(role_files: Vec<String>, summary_file: String, verdict_file: String) -> String
}
```

**Responsibilities:**
- List all generated files in "Files reviewed" field
- Include summary and verdict files in memory entry
- Maintain existing review_memory.md format

## Data Models

### File Naming Model

```rust
struct ReviewFile {
    prefix: u8,           // 00-99
    slug: String,         // e.g., "summary", "swe-review", "verdict"
    extension: String,    // always "md"
}

impl ReviewFile {
    fn filename(&self) -> String {
        format!("{:02}-{}.{}", self.prefix, self.slug, self.extension)
    }
}
```

**File Naming Rules:**
- Summary: `00-summary.md`
- Core roles: `01-swe-review.md`, `02-sa-review.md`, `03-qa-review.md`
- Triggered roles: `04-pe-review.md`, `05-se-review.md`, `06-oe-review.md`, `07-ceo-review.md`, `08-devil-advocate.md`
- Verdict: `99-verdict.md`

### Summary Content Model

```rust
struct SummaryContent {
    summary_paragraph: String,
    file_walkthrough: FileWalkthroughTable,
    risk_assessment: RiskAssessment,
    review_effort: ReviewEffort,
    table_of_contents: TableOfContents,
}

struct FileWalkthroughTable {
    rows: Vec<FileWalkthroughRow>,
}

struct FileWalkthroughRow {
    file_path: String,
    change_type: ChangeType,  // New, Modified, Deleted, Renamed, Config
    what_changed: String,
    notes: String,
}

struct RiskAssessment {
    level: RiskLevel,  // Low, Medium, High
    justification: String,
}

struct ReviewEffort {
    score: u8,  // 1-5
    explanation: String,
}

struct TableOfContents {
    links: Vec<TOCLink>,
}

struct TOCLink {
    role_name: String,
    filename: String,
}
```

### Role Content Model

```rust
struct RoleContent {
    role: Role,
    findings: Vec<Finding>,
}

struct Finding {
    severity: Severity,  // BLOCKER, MAJOR, SUGGESTION, NIT, QUESTION
    file_path: Option<String>,
    line_number: Option<usize>,
    description: String,
}

enum Role {
    SeniorSoftwareEngineer,
    SoftwareArchitect,
    QualityAssurance,
    PerformanceEngineer,
    SecurityEngineer,
    OperationsEngineer,
    CEO,
    DevilsAdvocate,
}

enum Severity {
    Blocker,
    Major,
    Suggestion,
    Nit,
    Question,
}
```

### Verdict Content Model

```rust
struct VerdictContent {
    suggested_tests: Vec<SuggestedTest>,
    recommendations: Vec<Recommendation>,
    verdict: Verdict,
    ai_caveat: String,
    summary_link: String,
}

struct SuggestedTest {
    name: String,
    test_type: TestType,  // unit, integration, e2e
    scenario: String,
    input_condition: String,
    expected: String,
    catches: String,
}

struct Recommendation {
    severity: Severity,
    description: String,
}

enum Verdict {
    Ship,
    ShipWithChanges,
    NeedsRework,
    NeedsDiscussion,
}
```

## Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of a system—essentially, a formal statement about what the system should do. Properties serve as the bridge between human-readable specifications and machine-verifiable correctness guarantees.*

### Property 1: File naming convention compliance

*For any* generated review output directory, all review files should have `.md` extension, follow the `<NN>-<slug>.md` naming pattern where NN is a two-digit number, and sort alphabetically in the correct reading order (summary first, verdict last).

**Validates: Requirements 1.1, 3.2, 6.1, 6.2, 6.3**

### Property 2: Summary file completeness

*For any* generated summary file, it should contain all required sections: a summary paragraph, file walkthrough table, risk assessment with justification, review effort score [1-5] with explanation, and a table of contents linking to all generated role files.

**Validates: Requirements 2.2, 2.3, 2.4, 2.5, 2.6**

### Property 3: Role file creation matches activation

*For any* review, role files should be created if and only if the corresponding role is active according to calibration rules—core roles (SWE, SA, QA) always create files, and triggered roles (PE, SE, OE, CEO, Devil's Advocate) create files only when their trigger conditions are met.

**Validates: Requirements 3.1, 3.4, 3.5**

### Property 4: Role file structure compliance

*For any* generated role file, it should begin with a level-1 header naming the role perspective, use only valid comment severity labels ([BLOCKER], [MAJOR], [SUGGESTION], [NIT], [QUESTION]), and cite specific file paths and line numbers when referencing code.

**Validates: Requirements 4.1, 4.3, 4.4**

### Property 5: Verdict file completeness

*For any* generated verdict file, it should contain the suggested tests section with proper format, prioritized recommendations organized by severity, one of the valid verdict values (Ship / Ship with changes / Needs rework / Needs discussion), and the AI review caveat text.

**Validates: Requirements 5.2, 5.3, 5.4, 5.5**

### Property 6: Triggered role numbering order

*For any* review where multiple triggered roles are active, the role files should be numbered 04-08 in this exact order: Performance Engineer (04), Security Engineer (05), Operations Engineer (06), CEO (07), Devil's Advocate (08).

**Validates: Requirements 6.4**

### Property 7: Markdown format compliance

*For any* generated review file, the content should be valid Markdown with proper syntax for headers, lists, code blocks, tables, and links.

**Validates: Requirements 1.2**

### Property 8: No unwanted file formats

*For any* review output directory, there should be no `.docx` files present unless the user explicitly requested that format.

**Validates: Requirements 1.3**

### Property 9: Directory integration

*For any* review, all output files should be written to the same timestamped directory that contains review_context.xml and review_prompt.md, using the same YYYYMMDD-HHMMSS timestamp format, and existing files in that directory should remain unmodified.

**Validates: Requirements 9.1, 9.2, 9.3, 9.4**

### Property 10: Review memory file references

*For any* review memory entry, the "Files reviewed" field should list all generated role files, the summary file, and the verdict file.

**Validates: Requirements 10.1, 10.3**

### Property 11: Directory creation resilience

*For any* review output path, if the directory does not exist, it should be created before writing files, and if any individual file creation fails, the remaining files should still be created successfully.

**Validates: Requirements 11.3, 11.4**

### Property 12: Cross-reference link format

*For any* link between review files, it should be a relative Markdown link (e.g., `[text](filename.md)`) rather than an absolute path, and the verdict file should include a link back to the summary file.

**Validates: Requirements 12.2, 12.4**

## Error Handling

### File Creation Errors

**Scenario:** File system errors during file creation (permissions, disk space, path issues)

**Strategy:**
- Log the error with full context (file path, error message)
- Continue creating remaining files
- Report partial success to user with list of failed files
- Do not fail the entire review process

**Example:**
```
Error creating 04-pe-review.md: Permission denied
Continuing with remaining files...
✓ Created 05-se-review.md
✓ Created 99-verdict.md
Review complete with 1 file creation failure
```

### Directory Creation Errors

**Scenario:** Cannot create output directory

**Strategy:**
- Attempt to create parent directories recursively
- If creation fails, report error and abort (cannot proceed without directory)
- Provide clear error message with path and suggested fixes

**Example:**
```
Error: Cannot create directory .materials/20250115-143022
Reason: Permission denied on .materials/
Suggestion: Check directory permissions or run with appropriate access
```

### Invalid Content Errors

**Scenario:** Generated content contains invalid Markdown or missing required sections

**Strategy:**
- Validate content before writing to file
- If validation fails, log warning but write file anyway (partial output better than none)
- Include validation warnings in file as comments
- Report validation issues to user

**Example:**
```markdown
<!-- Warning: File walkthrough table validation failed -->
<!-- Expected columns: File, Change type, What changed, Why / Notes -->

[content written anyway]
```

### Missing Role Content

**Scenario:** A triggered role has no findings to report

**Strategy:**
- Create the role file anyway (maintains consistent structure)
- Include a brief statement: "No issues found from this perspective"
- This is expected behavior, not an error

### Existing File Conflicts

**Scenario:** Review output files already exist in the target directory

**Strategy:**
- Check for existing review output files before starting
- If found, prompt user for action: overwrite, append timestamp, or abort
- Default behavior: append timestamp to avoid data loss
- Never silently overwrite existing review files
