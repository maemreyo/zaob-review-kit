# Multi-File Review Output Examples

This document provides complete examples of the multi-file review structure, showing both minimal and full review scenarios. Use these examples as templates when generating structured code reviews.

## Directory Structure

All review files are written to `.materials/<timestamp>/` alongside the existing prep-review files:

```
.materials/20250115-143022/
├── review_context.xml          # generated by prep-review
├── review_prompt.md            # generated by prep-review
├── review.patch                # generated by prep-review (commit mode)
├── temp/                       # agent working directory
│   ├── file_list.txt
│   ├── draft_findings.md
│   └── context_notes.md
├── 00-summary.md               # overview and navigation
├── 01-swe-review.md            # Senior Software Engineer findings
├── 02-sa-review.md             # Software Architect findings
├── 03-qa-review.md             # Quality Assurance findings
├── 04-pe-review.md             # Performance Engineer (conditional)
├── 05-se-review.md             # Security Engineer (conditional)
├── 06-oe-review.md             # Operations Engineer (conditional)
├── 07-ceo-review.md            # CEO perspective (conditional)
├── 08-devil-advocate.md        # Devil's Advocate (conditional)
└── 99-verdict.md               # recommendations and final decision
```

```markdown
These findings are generated by an AI and should be treated as a structured starting point, not a definitive verdict. Before acting on any finding:

- **Verify before flagging** — confirm the issue exists in the actual code, not a compressed or truncated representation of it
```

```markdown
# Senior Software Engineer Review

## Correctness and Edge Cases

`[BLOCKER]` **src/auth/jwt.rs:67** — Token validation doesn't check the `exp` claim before verifying signature. If signature verification is expensive, this allows attackers to force expensive operations with expired tokens. Check expiry first.

`[MAJOR]` **src/auth/jwt.rs:89** — Refresh token rotation not implemented. Once a refresh token is issued, it never expires. This violates security best practices and creates a permanent authentication backdoor if a refresh token is compromised.

`[SUGGESTION]` **src/auth/middleware.rs:34** — Error handling returns 401 for all auth failures (missing token, invalid token, expired token). Consider using 401 for invalid credentials and 403 for valid but insufficient permissions.

## Error Handling

`[MAJOR]` **src/handlers/auth.rs:56** — Database errors during login are returned as 500 with full error details. This leaks internal implementation details. Return generic error messages to clients and log details server-side.

`[NIT]` **src/auth/jwt.rs:45** — Unwrap used on token encoding. While unlikely to fail, consider returning Result for consistency and better error messages.

## Code Style

`[NIT]` **src/auth/jwt.rs** — Module could benefit from doc comments explaining the JWT flow, token structure, and security considerations.
```

### 02-sa-review.md (Full)

```markdown
# Software Architect Review

## Design Patterns

`[MAJOR]` **src/auth/jwt.rs** — JWT secret is loaded from config on every token operation. Consider loading once at startup and storing in application state to avoid repeated config access and potential inconsistency if config changes at runtime.

`[SUGGESTION]` **src/auth/middleware.rs** — Middleware extracts and validates tokens but doesn't cache validation results. For high-traffic endpoints, consider adding a short-lived in-memory cache (with proper invalidation) to avoid repeated signature verification.

## Separation of Concerns

`[QUESTION]` **src/handlers/auth.rs** — Login handler now contains JWT-specific logic (token generation). Consider moving this to the auth module to keep handlers focused on HTTP concerns and make the auth logic reusable.

## Architectural Consistency

`[MAJOR]` **Overall architecture** — The change introduces JWT tokens but doesn't remove the old session-based code. This creates two parallel authentication systems. Either complete the migration or document why both systems need to coexist.

## Scalability

No issues found. JWT-based auth is stateless and scales horizontally better than session-based auth.
```

### 04-pe-review.md (Full)

```markdown
# Performance Engineer Review

## Database Queries

`[MAJOR]` **src/auth/middleware.rs:45** — Middleware queries database on every request to validate refresh token. This adds a DB round-trip to every authenticated endpoint. Consider:

- Using only access tokens for request auth (no DB lookup)
- Caching token validation results with short TTL
- Using Redis for token blacklist instead of DB queries

## Algorithmic Complexity

`[SUGGESTION]` **src/auth/jwt.rs:67** — Token signature verification uses RS256 (RSA). Consider HS256 (HMAC) for better performance if you don't need public key verification. RSA verification is ~10x slower than HMAC.

## Memory Allocation

No significant issues found. Token operations allocate reasonably.

## Blocking Operations

`[QUESTION]` **src/auth/middleware.rs:45** — Database query in middleware is synchronous. Confirm this is using async DB driver to avoid blocking the async runtime.
```

### 05-se-review.md (Full)

```markdown
# Security Engineer Review

## Authentication & Authorization

`[BLOCKER]` **src/config.rs:23** — JWT_SECRET defaults to "change-me-in-production" if not set. This is a critical security vulnerability. Application should refuse to start if JWT_SECRET is not configured or is set to the default value.

`[BLOCKER]` **src/auth/jwt.rs:89** — Refresh tokens stored in database as plaintext. If database is compromised, all refresh tokens are exposed. Hash refresh tokens before storage (similar to password hashing).

`[MAJOR]` **src/auth/jwt.rs:34** — Token expiry set to 24 hours. This is too long for an access token. Recommend 15 minutes for access tokens, longer expiry for refresh tokens.

## Input Validation

`[SUGGESTION]` **src/handlers/auth.rs:78** — Refresh token endpoint doesn't rate limit. Attackers could brute-force refresh tokens. Add rate limiting per IP and per user.

## Secrets & Credentials

`[BLOCKER]` **src/auth/jwt.rs:12** — JWT secret loaded from environment variable but no validation of secret strength. Require minimum length (32 bytes) and entropy check.

`[MAJOR]` **src/handlers/auth.rs:56** — Login endpoint doesn't implement account lockout after failed attempts. Vulnerable to brute-force password attacks.

## Dependencies

`[SUGGESTION]` **Cargo.toml** — jsonwebtoken crate version is "0.8". Check for known CVEs and consider updating to latest version.
```

### 08-devil-advocate.md (Full)

```markdown
# Devil's Advocate

## Challenge Assumptions

`[QUESTION]` **Necessity** — Why replace session-based auth? The PR description mentions "modernization" but doesn't cite specific problems with the current system. If sessions work fine, this change adds risk without clear benefit.

`[QUESTION]` **JWT benefits** — JWTs are often chosen for stateless auth, but this implementation queries the database on every request to validate refresh tokens. This eliminates the statelessness benefit while adding JWT complexity.

## What Could Go Wrong

`[MAJOR]` **Secret rotation** — JWT_SECRET rotation forces all users to re-login. In a large system, this could cause a stampede of login requests. Has this scenario been load tested?

`[MAJOR]` **Token revocation** — JWTs can't be revoked without maintaining a blacklist (which requires state). If a user's account is compromised, their tokens remain valid until expiry. Session-based auth can revoke immediately.

`[QUESTION]` **Mobile app impact** — Mobile apps will need updates to handle token-based auth. What's the plan for users who don't update their apps? Will old app versions stop working?

## Alternative Approaches

`[SUGGESTION]` **Hybrid approach** — Consider keeping session-based auth for web clients and adding JWT only for API clients that need stateless auth. This avoids breaking existing web flows while enabling new use cases.

`[SUGGESTION]` **Incremental migration** — Instead of replacing auth system in one PR, consider:

1. Add JWT support alongside sessions (both work)
2. Migrate clients gradually
3. Remove session support after all clients migrated
   This reduces risk and allows rollback at each step.
```

## Key Patterns and Guidelines

### File Naming Convention

- **Summary:** `00-summary.md` (always first)
- **Core roles:** `01-swe-review.md`, `02-sa-review.md`, `03-qa-review.md` (always present)
- **Triggered roles:** `04-pe-review.md`, `05-se-review.md`, `06-oe-review.md`, `07-ceo-review.md`, `08-devil-advocate.md` (conditional)
- **Verdict:** `99-verdict.md` (always last)

### Severity Labels

Use exactly one label per finding:

- `[BLOCKER]` — Incorrect behavior, security hole, data loss risk (blocks merge)
- `[MAJOR]` — Significant design or logic flaw, hard to fix later (blocks merge)
- `[SUGGESTION]` — Better approach exists, but current code works (doesn't block)
- `[NIT]` — Style, naming, minor readability (doesn't block)
- `[QUESTION]` — Asking for clarification, not a criticism (doesn't block)

### File References

Always include file path and line number when citing code:

```markdown
`[BLOCKER]` **src/auth/jwt.rs:67** — Description of the issue
```

### Cross-References

Use relative Markdown links to reference other review files:

```markdown
See also the [Security Engineer review](05-se-review.md) for related concerns about token storage.
```

### Suggested Tests Format

Each suggested test must include:

```markdown
- **test_name** (type: unit | integration | e2e)
  Scenario: One sentence describing what this test exercises
  Input / condition: Specific values, state, or trigger
  Expected: What should happen — return value, side effect, error
  Catches: What bug or regression this prevents
```

### Table of Contents Format

In summary file, link to all generated role files:

```markdown
## Table of Contents

- [Software Architect Review](02-sa-review.md)
```

### Verdict Back-Link

Always include link back to summary at the top of verdict file:

```markdown

```

## When to Use Minimal vs Full Review

**Minimal Review** (SWE, SA, QA only):

- Small changes (< 50 lines)
- No security-sensitive code
- No database or performance concerns
- No new dependencies
- No breaking changes

**Full Review** (all triggered roles):

- Large changes (> 300 lines)
- Auth, security, or permission code
- Database queries or schema changes
- New dependencies
